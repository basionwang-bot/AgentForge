> **本模块教学目标：** 学会写"第一次就能跑通"的指令，把返工次数降到最低。

---

## 一个朴素的真相

你交代得越模糊，它就越得靠猜。猜错了，你就得返工；返工时那些失败的尝试，又一张张堆上了它的写字台。台子越堆越满，它越来越糊涂，于是越改越乱——恶性循环就这么转起来了。

> **提问的精度，直接决定返工的次数。** 花 30 秒把话说清楚，能省下半小时的来回扯皮。

这一篇就教你怎么"把话说清楚"。

---

## 四个改写模板（看前后对比，秒懂）

下面四种情况，覆盖了你日常九成的提问。每一种都给你一个"从模糊到精确"的改写示范，照着套就行。

### 策略 1：把范围圈小

| | |
|---|---|
| ❌ 模糊 | "给 foo.py 加测试" |
| ✅ 精确 | "给 foo.py 写测试，**覆盖用户没登录时的情况**，**别用 mock 那套替身**" |

**为什么更好？** "加测试"它能有一百种理解。你一圈定"测哪个场景、用不用某个手法"，它就没得乱猜了。

### 策略 2：告诉它"去哪儿找答案"

| | |
|---|---|
| ❌ 模糊 | "这个 API 怎么这么奇怪？" |
| ✅ 精确 | "**翻一下 ExecutionFactory 的改动历史**，总结这个接口是怎么一步步演变成现在这副样子的" |

**为什么更好？** 你直接给它指了路——"答案藏在改动记录里"，它就不用瞎蒙，跑去翻一遍，给你一份有理有据的来龙去脉。

### 策略 3：递给它一个"照着抄"的样板

| | |
|---|---|
| ❌ 模糊 | "加一个日历组件" |
| ✅ 精确 | "**先看首页现有的部件是怎么做的，HotDogWidget.php 就是个好样板**，照这个路子做一个日历组件，**别引入新的外部依赖**" |

**为什么更好？** 这是新手最容易忽略、却最管用的一招。给它一个范本照着抄，做出来的东西立刻和你项目原来的风格对得上，而不是它自顾自另起炉灶发明一套。

### 策略 4：只描述"哪里不舒服"，别替它猜病因

| | |
|---|---|
| ❌ 模糊 | "修一下登录 bug" |
| ✅ 精确 | "用户反馈**挂机太久、登录就失败**。去 `src/auth/` 里查管登录凭证刷新的那段逻辑。**先写一个能稳定复现这毛病的测试，再动手修**" |

**为什么更好？** 你把"什么情况下、出了什么毛病、大概在哪、怎么算修好"都讲清楚，它就能像个好大夫一样直奔病根，而不是头痛医头、瞎试一通。

> 📌 看出规律了吗？四个模板的共同点都是：**把那些"你脑子里默认知道、却没说出口"的信息，明明白白写出来。** 你觉得理所当然的背景，它一概不知道。

## 五种"喂料"方式（怎么把信息递到它手上）

光会问还不够，你还得知道怎么把资料塞给它。一共五种：

1. **`@文件名`** —— 直接点名一个文件，它会先读再答。
   ```text
   照着 @src/components/Button.tsx 的写法，做一个 Card 组件
   ```
2. **截图 / 图片** —— 直接粘贴或拖进去。报错的截图、设计稿、表格照片都行，它都看得懂。
3. **网页链接** —— 把网址甩给它让它读。常上的几个网站可以用 `/permissions` 加进"放行名单"，省得它每次都回头问你"能去看吗"。
4. **把命令结果直接灌给它** —— 一行命令把输出喂进去：
   ```bash
   cat error.log | claude
   ```
5. **让它自己去取** —— 直接告诉它"自己跑命令 / 用工具 / 读文件"去把需要的信息找齐。比如："跑 `git log` 看看最近都改了啥。"

新手建议：**先把 `@文件名` 和"贴截图"这两招练熟**，日常八成的场景就够用了。

## 🔥 杀手锏：让 Claude 反过来"面试"你

这是**对不懂技术的新手最友好的一招**，请重点掌握。

很多时候你心里有个模糊的想法，但说不清楚到底要什么——这太正常了。这时候**别硬憋着自己写需求**，反过来，请它当面来采访你。想象一下你请了个经验老到的产品经理坐你对面，一个问题一个问题把你脑子里没想清的地方挖出来：

```text
我想做 [一句话描述]。用 AskUserQuestion 工具详细采访我。
问我技术实现、UI/UX、边界情况、顾虑和取舍。
不要问显而易见的问题，挖我没想到的难点。
问完之后，把完整的需求规格写进 SPEC.md。
```

它就会像那位产品经理一样，**一个一个问题追着你问**，把你自己都没想清楚的角落全挖出来。问完，它顺手帮你把这些整理成一份 `SPEC.md`，一份白纸黑字的需求清单。

**然后是关键一步：写完 SPEC.md，另起一个全新的对话去执行它。**

为什么要重开？因为刚才那场采访，来来回回聊了一大堆，早把它的写字台堆满了。新对话桌面干干净净，而你手里又攥着一份能反复翻看的需求清单——这比一段乱糟糟的聊天记录可靠太多了。

> 💡 什么样的需求清单最顶用？**自己能讲全自己**的：写明会动到哪些文件和接口、哪些事**不归这次管**、最后拿什么**从头到尾跑一遍的检验**来证明功能真的能用。

## 等等——什么时候反而该"故意问得模糊"？

不是所有时候都要抠得精确。**在探路阶段，模糊提问反而更好。**

当你想看看它**会怎么理解一个问题**、想借它的眼睛挖出你自己想不到的点子时，就故意把话问得开一点：

```text
这个文件，你觉得有什么可以改进的地方？
```

这种开放的问法能让它**放开手脚发挥**，经常能给你翻出一些你压根没注意到的问题。

**一句口诀：** 想要稳稳的结果 → 把话问准；想要意外的灵感 → 把话问松。

## ✍️ 学员练习

挑一件你最近想让 AI 帮你做的事，用本篇的方法做两遍：

1. 先按你的老习惯，随手写一句话指令。
2. 再用"四个模板"改写一遍（圈小范围 / 指路找答案 / 递个样板 / 描述症状，挑用得上的）。
3. 把两次的产出摆一起，看看差多少。

然后试一次**杀手锏**：让它用 AskUserQuestion 采访你，生成一份 SPEC.md，再开个新对话去执行。亲自体会那种"需求一下子就清晰了"的感觉。
