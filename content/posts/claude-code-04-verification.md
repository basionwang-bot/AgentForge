> **本模块教学目标：** 让 Claude 自己检查自己。这是"帮手"和"靠谱员工"的真正分水岭。

---

## 先问个扎心的问题：你怎么知道它干对了？

新手的日子，多半是这么过的：你让它干活 → 它喜滋滋回你一句"完成啦！" → 你将信将疑，**自己跑一遍、自己看一遍**，发现有毛病 → 你打回去让它改 → 它又喜滋滋来一句"完成啦！" → 你又得自己验一遍……如此循环。

看出问题在哪了吗？**你活生生把自己熬成了一台人肉质检机。** 它埋的每一个雷，都得等你亲自一脚踩上去才发现。这样的活儿，你哪敢走开半步？端着咖啡都不踏实。

说到底，毛病就出在一句话上：

> **给它一把它自己就能用的尺子（跑测试、跑构建、截图比对），让它自己量量干得对不对。手里没这把尺子，"看起来好像做完了"就是它能给你的唯一信号——而这个信号，靠不住。**

一旦把这把尺子塞进它手里，差别立马就摆出来了：

| | 没尺子 | 有尺子 |
|---|---|---|
| 它说"完成"时 | "我觉得写好了"（信不过） | "测试全过了，输出在这"（信得过） |
| 你的角色 | 人肉质检机，走不开 | 主管，可以放心走开 |
| 错误啥时候被发现 | 等你亲自一脚踩上 | 它自己当场逮住、当场修掉 |

**这道坎，就是"你得盯着的活"和"你能走开的活"之间的那道坎。** 而"能走开"这三个字，正是你将来迈进模块七(自动化)的入场券——没它，后面全免谈。

---

## 三个改写示范（怎么把尺子递到它手里）

### 示范 1：把"怎么才算对"明明白白写出来

| | |
|---|---|
| ❌ 之前 | "实现一个邮箱校验函数" |
| ✅ 之后 | "写 `validateEmail` 函数。**测试用例：`user@example.com` 为 true，`invalid` 为 false，`user@.com` 为 false。实现后跑测试。**" |

你把"怎样才算对"白纸黑字一条条列清楚，它就能自己跑、自己核对——这把尺子，你已经稳稳递到它手上了。

### 示范 2：界面的活，给它一把"用眼睛量"的尺子

| | |
|---|---|
| ❌ 之前 | "把仪表盘做好看点" |
| ✅ 之后 | "[贴设计图] 照这个实现。**完成后截图，和原图对比，列出差异并修复。**" |

"好看"是句没法量的主观话，一千个人有一千个标准；而"截个图、跟设计图摆一块儿比一比、把不一样的地方挑出来"，这才是它自己就能动手量的硬尺子。

### 示范 3：治病要治根，别光抹个表面

| | |
|---|---|
| ❌ 之前 | "构建失败了" |
| ✅ 之后 | "构建报这个错：[贴报错]。修复它并**验证构建通过**。要**解决根本原因，不许把报错压掉**。" |

最后那句话千万别省——一旦省了，它为了让那行碍眼的红字赶紧消失，很可能干脆把检查关了、把报错偷偷吞了。表面上是清清爽爽了，病根却还烂在肚子里，过两天照样发作。

## 验证的四个层级（从最轻到最重）

不是每个活儿都得搬出最重的家伙，得看菜下饭、量体裁衣：

1. **就在话里顺手带一句（最轻）** —— 在同一条消息里随口加一句："**跑检查，不过就接着改，改到过为止。**" 日常最常用，本质上就是多敲一行字的事，零成本。

2. **整个会话盯着一个目标** —— 用 `/goal` 给它定一条达标线。它每干完一轮，就有个不掺和具体活、专门挑刺的"评审"来复查一遍，没达标就接着返工。适合那种要反复打磨、必须达到某个标准才肯罢休的活。

3. **过不了关就别想收工（硬卡死）** —— 用 **Stop hook** 挂上你的检查脚本：**检查一天不通过，就一天不许它结束这回合。** 这是"铁规矩"级别的兜底（连着被拦 8 次之后系统会松手放行，免得它在死循环里出不来）。具体怎么配，模块六、模块七会手把手讲。

4. **专门请个第三方来挑刺（最重）** —— 派一个**桌子干干净净、对前因后果一无所知**的助手，专门来当那个唱反调的。

   > 为什么非得找个"啥都不知情"的？因为**自己干的活，自己打分总会下意识手软。** 刚写完代码那个会话，心里护着自己的产出，总觉得"我亲手写的能有啥问题"。换上一双没有任何成见、谁也不认识的新眼睛，才挑得出真毛病。

   给新手的建议：**先把第 1 层用顺手再说。** 养成每次都顺嘴补一句"跑测试，改到通过"的习惯，光这一下，你的体验就立马蹿上一个档次。

---

## 📌 配套要求：让它"拿证据来"，别光动嘴说成功

这一招很多人随手忽略了，太可惜。别轻信"我做完了"这种空口白牙的话，要它**把证据当面摆出来**：

- 测试跑出来的真实输出（是真跑出来的结果，不是它嘴上说"应该能过"）
- 到底跑了哪条命令、返回了啥
- 最终结果的截图

```text
完成后，把你跑的命令和实际输出贴给我，别只说"做好了"。
```

**这么做有两个实打实的好处：**

1. 你瞄一眼现成的证据，比自己吭哧吭哧重跑一遍**快太多了**。
2. 这招对你**人不在场、没盯着的活**一样灵——回头一看证据摆在那儿，靠不靠谱一目了然，不用猜。

---

## ✍️ 学员练习

挑一个你平时常交给它的活儿，亲手给它改造成"带尺子"的版本：

1. 你原来的指令是怎么写的？（八成压根没提"怎么才算对"）
2. "怎样才算干对了"？把这条标准改写成它**自己就能动手量的检查**（测试用例 / 构建通过 / 截图比对）。
3. 在指令末尾稳稳补上一句"跑检查，改到通过，完成后把输出贴给我"。

跑这么一次，好好体会一下那种"我总算不用自己一遍遍验了"的解脱感。

---

## 本模块一句话总结

> **没尺子的活，你永远走不开。** 给它一把自己能用的尺子，再要它把证据摆出来——这就是从"AI 帮手"升级成"AI 员工"的那道坎。
