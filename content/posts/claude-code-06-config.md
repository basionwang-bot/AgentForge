> **本模块教学目标：** 给你（或你的团队）的 Claude Code 布置一个顺手的工位。前面五个模块教"怎么用得好"，这一模块教"怎么让它一上来就好用"。

---

## 为什么要配置环境？

想象一下：你新来了个实习生，脑子挺灵，可每天一上班，你都得把"我们这儿的规矩"从头絮叨一遍——讲到第十天，你嗓子都哑了，它还时不时忘。

环境配置，干的就是省你这口气的事:**把那些"天天要叮嘱的话"一次性贴好、装好，让它默认就照着做。** 配一次，往后每回开工自动生效。这就是"省力"的杠杆——前面学的那些技巧，不用你每次再亲自上手施展一遍了。

这一模块工具有点多，新手别慌、更别想着一口气全上。**先把前两个吃透——一张便利贴、一个安检员，剩下的等用到再回头看。**

## 6.1 CLAUDE.md —— 贴在工位上的"便利贴守则"

这是最重要的一个配置，**新手必学。**

它是什么？说白了，就是一张**贴在工位上的便利贴守则**。你把"这个项目的规矩"写在上头，它每天一上班、每开一个新对话，**先扫一眼这张便利贴**，然后才动手。这么一来，你就不用每次都把那套话重讲一遍了。

怎么贴上去：敲一个 **`/init`**，它会先把你的项目里里外外打量个遍，自动给你写一份初稿。然后——这才是关键——你得**一遍遍打磨**它，越改越顺手。

### ⚠️ 最重要的一条原则：越短越好

> **便利贴写得越长、越啰嗦，它越当没看见。** 这是新手最常踩的坑。

你想啊：门口贴一张三句话的守则，谁路过都记得住;贴一张三页纸的小作文，没人看到第二段就走了。机器也一个脾气。

判断某一行该不该留，就问自己一句:**"把这行删了，它会犯错吗？不会？那就删。"**

下面这张表是黄金标准——**照着它写，别越界:**

| ✅ 该贴上便利贴 | ❌ 别往上贴 |
|---------------------|------------|
| 它猜不到的 Bash 命令（比如某个特殊的构建命令） | 它扫一眼代码就能看出来的东西 |
| 和默认不一样的代码风格 | 语言通用规范（它本来就懂） |
| 测试命令和你的偏好 | 大段 API 文档（给个链接就够了） |
| 分支命名、PR 规范 | 三天两头就变的信息 |
| 项目特有的架构决定 | "要写干净的代码"这种正确的废话 |
| 环境怪癖（某个少了就跑不起来的环境变量） | 一个文件一个文件地解说代码库 |

### 几个实用技巧

- 内容太多，可以用 `@路径` **把别的文件牵进来**——把细节拆出去单放，便利贴本身保持清清爽爽。
- 真想让它当回事的规则，前面加一句 **"IMPORTANT"** 或 **"YOU MUST"**，就像拿红笔圈了个重点，它会更上心。
- **这张便利贴贴在哪？** 看你想让谁看到:
  - `~/.claude/CLAUDE.md` —— 贴在你自己的"总工位"上，你手头所有项目都看得到
  - 项目根目录的 `CLAUDE.md` —— 进 git，**全队共用这一张**
  - `CLAUDE.local.md` —— 只给你自己看的私房话，记得加进 `.gitignore`，别脏了团队那张
  - 大型项目（monorepo）的父子目录里，它**走到哪个文件夹、就自动翻到哪张**对应的便利贴

## 6.2 权限三件套 —— 少点几次烦人的"点点点"

新手很快会撞上一个烦恼:它每走一步都弹个窗问你"这条命令能跑吗?"，问得你点鼠标点到手指发酸。三个法子治它：

1. **Auto mode（自动模式）** —— 想象机场安检口站了个**安检员**：每条命令它替你先过一遍，**只拦看着危险的**（越权的、动了不明来路设施的、像是被坏内容带跑偏的那种可疑动作），正常的，挥挥手就放行。
2. **`/permissions` 白名单** —— 把那些**你闭着眼也知道安全**的命令登记一下，往后这几条直接放行、再不来烦你。比如 `npm run lint`、`git commit`。
3. **`/sandbox` 沙箱** —— 给它圈一块**游乐场围栏**：在操作系统这一层，把文件和网络都隔起来，围栏里随它怎么蹦跶，就是越不出那条边界。

新手怎么上手:**先把你天天用、又绝对安全的那两三条命令，用 `/permissions` 登记上**，耳根立刻清净一大半。

## 6.3 Skills（技能）—— 按需取用的"工具箱抽屉"

**想搞懂 Skills，关键看它和便利贴怎么分工:**

- **便利贴（CLAUDE.md）** —— 每次开工都要扫一眼。所以上面只放**人人都用得上的普适规则**(放杂了，整张都没人细看)。
- **Skills（技能）** —— 平时收在抽屉里，眼不见、不占地方;**碰上那一类活，才拉开对应的那个抽屉**。所以专门塞"特定时候才用得上"的领域知识和流程。

怎么造一个抽屉:在 `.claude/skills/技能名/SKILL.md` 里写好。该用的时候它自己拉开，你也可以敲 `/技能名` 亲手把它叫出来。

举个例子，一个装着"API 规范"的抽屉:

```markdown
---
name: api-conventions
description: 我们服务的 REST API 设计规范
---
# API 规范
- URL 路径用 kebab-case
- JSON 属性用 camelCase
- 列表接口必须带分页
- 版本号放 URL 路径（/v1/、/v2/）
```

这样,只有当它真的在碰 API 的时候,这个抽屉才拉开、规范才进场;别的时候它老老实实收着,谁也碍不着。

## 6.4 Hooks（钩子）—— 说一不二的自动机关

理解 Hook，先记牢它和便利贴的**根本区别:**

> **便利贴上的话是"口头叮嘱"——它大概率照做，但偶尔也会走神忘了。Hook 是装在门口的"绊线机关"——每次有人经过,必然绊响,一次不漏。**

什么意思?比方说你想要"每次改完文件就跑一下 eslint 检查"。写在便利贴上,它**可能**哪天就忘了;做成 Hook,就等于在门口拉了根绊线——只要有人(某个动作)经过,**必然**触发,根本不指望它自觉。

好消息:**这机关你不用自己研究怎么装,张嘴让 Claude 帮你装就行:**

```text
写一个 hook，每次改完文件自动跑 eslint
```

想瞧瞧自己门口都装了哪些机关,敲一个 **`/hooks`** 看一眼。(具体怎么装、该装哪些,模块七给你七个"拿来即用"的现成配方。)

## 6.5 MCP —— 接上外部世界的"电话线"

MCP，就像给它**接上几根通往外部世界的电话线**:这头连 Notion，那头连 Figma，还能往数据库、监控系统上搭线……线一接通,它就不再是闷头改代码的"宅男"了,你那一整圈工具,它都能打个电话过去聊两句。

用 `claude mcp add` 来接线。接通之后它能干嘛?

- 照着一张 issue 单,直接把功能给你实现出来
- 钻进数据库翻东西查数据
- 扒一把监控数据做分析
- 对着 Figma 设计稿,把界面照着画出来

新手了解个大概就行,等你哪天冒出个具体念头(比如"我想让它直接读我 Notion 里写的需求")，再回来接这根线也不迟。

## 6.6 自定义子代理 + 插件

- **自定义子代理**:在 `.claude/agents/` 里养几个**专职助手**(比如一个专挑安全漏洞的"审查员")。每个助手都坐在**自己单独的房间、用自己单独的那套工具权限**,你还能点名它用哪个模型。派活给它,它进自己房间闷头忙,回来只递给你一张结论,过程不占你的桌面。
- **插件**:用 `/plugin` 逛插件市场。**一个插件 = 技能 + 钩子 + 子代理 + MCP 打成一个包**,一键就装好一整套,省得你一样样去配。

## 6.7 一个省力小习惯：优先给它配 CLI 工具

给它装上 `gh`（GitHub CLI）、`aws`、`gcloud` 这类命令行小工具——**这是它对外打交道时最省"脑容量"的方式**:一条命令把事就办了,不用它吭哧吭哧去啃一大堆网页(读网页可是会把写字台堆得满满当当的)。

更妙的是,就算碰上没摸过的工具,它还能**当场现学现卖**:

```text
先跑 `foo --help` 学会这个工具，然后用它完成 A、B、C
```

## ✍️ 学员练习

今天就动手做两件事（新手版）：

1. 在你的项目里跑一次 `/init` 生成便利贴(CLAUDE.md)，然后**对照那张黄金标准表狠心删减**，删到每一行都"删了它就会犯错"为止。短，就是美。
2. 把你最常用、最安全的 2-3 条命令用 `/permissions` 登记上，从此跟"点点点"说拜拜。

至于进阶那几样（Skills / Hooks / MCP），等你真被某个痛点扎一下,再回来配也来得及。

## 本模块一句话总结

> **配置的本质,是把"每天要叮嘱的话"一次贴好。** 便利贴(CLAUDE.md)管"每次都要的规矩"(越短越好)，抽屉(Skills)管"按需取用的知识"，机关(Hooks)管"必须说一不二执行的动作"，电话线(MCP)管"连外面的世界"。光把前两个配利索了，你就已经甩开大多数人了。
