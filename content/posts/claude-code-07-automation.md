> **本模块教学目标：** 让 Claude 在你不在场的时候干活——这是"一个人，也是一支队伍"的字面实现，也是整门课的高潮。
>
> ⚠️ **前置提醒：** 本模块偏进阶。**没把模块四（验证闭环）吃透，别碰自动化。** 道理很简单:你人都不在场,出了岔子谁拦?唯一能兜底的,就是它自己那点自检本事。

## 自动化的正确"爬梯"顺序

别一上来就奔着"完全撒手、无人值守"去,那不是抄近道,那是抄近道翻进沟里。稳妥的进阶路线，像爬梯子，一级一级来：

> **先在交互模式下把流程跑顺 → 封装成 Skill → 接上 hook / 定时任务 → 最后才轮到完全无人值守。**

下面就顺着这架梯子，一级一级往上爬。

---

## 7.1 非交互模式 `claude -p` —— 一切自动化的地基

平时你敲 `claude`，是走进一间办公室、和它面对面聊。但自动化要的不是聊天,要的是:**递张纸条进去,它把结果递出来,人不用进门。** 这就是 `-p`（print 模式）干的事:

```bash
# 一次性提问
claude -p "解释这个项目是做什么的"

# 结构化输出（JSON），给脚本消费
claude -p "列出所有 API 端点" --output-format json

# 管道串联：Claude 的输出喂给下一个命令
claude -p "你的提示词" --output-format json | your_command

# 管道灌入：把数据喂给 Claude
cat error.log | claude -p "找出报错的根本原因"
```

正因为它就是一条普普通通的命令,**你能把它塞进任何脚本、CI 流水线、定时任务里**——哪儿都插得进去。这,就是后面一切自动化的地基。

**几个配套参数（这是你给它系的"安全绳"）：**

- `--allowedTools "Edit,Bash(git commit *)"` —— **划定它只能动哪几样工具**。没人盯着的时候,这根绳子兜着你。
- `--permission-mode auto` —— 让它自己跑,命令交给后台那位"安检员"过目。（注意:`-p` 模式下,要是安检员**一次次拦它**,它会干脆自动停手——因为现场没人能帮它拍板。）
- `--verbose` —— 调试时打开,看它每一步在干嘛;上了生产环境就关掉。

```bash
# 完整示例：全自动修 lint 错误
claude --permission-mode auto -p "fix all lint errors"
```

## 7.2 批量扇出（Fan-out）—— 一次处理两千个文件

要干一票大的——大规模迁移、大批量分析（比如"把 2000 个文件从 React 迁到 Vue"）？想象一条流水线:文件一个个丢进去,同一道工序挨个过。标准三步:

1. **先列任务清单** —— 让 Claude 把所有要处理的文件挨个列出来,存进 `files.txt`。这就是流水线上待加工的"料"。
2. **写个循环脚本** —— 对每一个文件,跑一遍 `claude -p`,像传送带把料一件件送上工位：
   ```bash
   for file in $(cat files.txt); do
     claude -p "把 $file 从 React 迁移到 Vue。只回答 OK 或 FAIL。" \
       --allowedTools "Edit,Bash(git commit *)"
   done
   ```
3. **先拿 2-3 个试水,再全量开跑** —— 别急着 2000 个一起上。先挑几个试,看哪儿翻车、回头**把提示词打磨顺**,满意了再放开。这一步千万别省——省了,搞不好就是 2000 个文件一起翻车,场面没法收。

## 7.3 Hooks 自动化配方（七个拿来即用）

前面说过,Hook 就是门口的"绊线机关":Claude 干活干到某个节点,它**必然**自动触发一条 shell 命令。在哪儿装这些机关?写进 `~/.claude/settings.json`（全局,所有项目通用）或 `.claude/settings.json`（只管当前项目）。下面七个配方,抄了就能用。

**① 改完文件，自动格式化一遍**（PostToolUse 事件）

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{ "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write" }]
    }]
  }
}
```

**② 它需要你拍板时，弹个桌面通知**（Notification 事件，macOS）——这样你尽管去忙别的,该叫你时它会喊你

```json
{
  "hooks": {
    "Notification": [{
      "matcher": "",
      "hooks": [{ "type": "command",
        "command": "osascript -e 'display notification \"Claude Code 需要你\" with title \"Claude Code\"'" }]
    }]
  }
}
```

matcher 还能再分细点:`permission_prompt`（它等你审批时）、`idle_prompt`（活干完了、等你下一步指令时）等等。

**③ 给敏感文件上把锁，不许它碰**（PreToolUse 事件 + 退出码 2 拦截）

写个小脚本 `.claude/hooks/protect-files.sh`,让它在 Claude 动手前先查一眼:这次要碰的路径,是不是 `.env`、`package-lock.json`、`.git/` 这些"碰不得"的名单里的?命中就 `exit 2`,当场拦下。Claude 会收到"为啥被拦",然后自己换个方案。

**④ 上下文被压缩后，自动把关键信息补回来**（SessionStart 事件，matcher 为 compact）

写字台堆满、被收拾过一轮之后,容易忘事。这个机关就负责把最要紧的几句再贴回它眼前：

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "compact",
      "hooks": [{ "type": "command",
        "command": "echo '提醒：用 Bun 不用 npm。提交前跑 bun test。当前迭代：认证重构。'" }]
    }]
  }
}
```

这里的 `echo` 可以换成任何会动态出结果的命令,比如 `git log --oneline -5`,把最近几条提交记录直接塞回它脑子里。

**⑤ 配置一改动，自动记一笔账**（ConfigChange 事件）—— 谁在会话里改了配置,记下来,合规留痕,日后好查。

**⑥ 进了某个目录、或文件一变，自动重新加载环境变量**（CwdChanged / FileChanged 事件，配合 direnv）。

**⑦ 特定的权限弹窗，自动点"同意"**（PermissionRequest 事件）—— 比如自动放行 ExitPlanMode,计划一出来就直接开干,不劳你点一下。

> 进阶:除了上面这种"死守一条命令"的机关,还有 **prompt-based hooks** 和 **agent-based hooks**——干脆请一个 Claude 模型来当门卫,那些**要动脑判断、不是一刀切死规则**的拦不拦,交给它现场拿主意。

## 7.4 Skills 作为"一键工作流"

把一整套固定流程,塞进一个抽屉,以后带个参数一叫就跑:

```markdown
---
name: fix-issue
description: 修复一个 GitHub issue
disable-model-invocation: true
---
分析并修复 GitHub issue：$ARGUMENTS

1. 用 `gh issue view` 获取 issue 详情
2. 理解 issue 描述的问题
3. 在代码库中搜索相关文件
4. 实现修复
5. 编写并运行测试验证
6. 确保通过 lint 和类型检查
7. 写清晰的提交信息
8. 推送并创建 PR
```

往后你只要敲一句 `/fix-issue 1234`,从看 issue 到提 PR,整条流水线自己一口气跑完。

那行 `disable-model-invocation: true` 是什么意思?**这种会动真格、有副作用的流程,只许你亲手按下启动键**,Claude 自己不准擅自调用它。一道保险。

## 7.5 定时任务：让它按时间表自动干活 ⭐

给它**定个闹钟**,到点自动干一件事。闹钟有三种定法,各有各的脾气：

| | Cloud 云端 | Desktop 桌面 | `/loop`（会话内） |
|---|---|---|---|
| 运行位置 | Anthropic 云端 | 你的电脑 | 你的电脑 |
| 需要开机 | 否 | 是 | 是 |
| 需要会话开着 | 否 | 否 | 是 |
| 重启后保留 | 是 | 是 | 否 |
| 访问本地文件 | 否（全新克隆） | 是 | 是 |
| 最小间隔 | 1 小时 | 1 分钟 | 1 分钟 |

**一句口诀帮你选：** 想要稳稳当当、人走了照跑,用**云端**;要碰你电脑上的本地文件,用**桌面**;只是会话里临时盯个进度,用 **`/loop`**。

**`/loop` 快速上手：**

```text
/loop 5m 检查部署是否完成，告诉我结果
/loop 每 2 小时检查一次构建状态
/loop 20m /review-pr 1234        ← 定时循环执行另一个技能
```

间隔单位支持 s / m / h / d,不写的话,默认每 10 分钟响一次。

**只想提醒一次的，张嘴说人话就行：**

```text
下午 3 点提醒我推 release 分支
45 分钟后检查集成测试过了没有
```

**管这些任务也一样省事：** 直接问"我现在有哪些定时任务？" / "把那个部署检查任务取消掉"。（底下其实是 CronCreate / CronList / CronDelete 三个工具在干活,用标准五段 cron 表达式,单个会话最多挂 50 个任务。）

**⚠️ 讲课必提的几个坑：**

- `/loop` 是**会话级**的:**终端一关,它就全没了**;循环任务挂满 7 天也会自动过期。
- 任务是在你两个回合**之间**触发的,Claude 正埋头忙的时候,它得排队等着。
- 时间按你**本地时区**算。
- 真要做到人走机器照跑、铁打不误的,请上 **Cloud 定时任务**或 **GitHub Actions 的 schedule 触发器**。

**事件驱动（比"定时问一遍"更聪明）：** 与其让它每隔几分钟跑去问一声"好了没"(轮询),不如用 **Channels**,让 CI、监控、聊天消息**主动推**进正在跑的会话——构建一挂,失败信息当场拍到它脸上,它立马就接招,不用干等下一次轮询。

## 7.6 CI/CD 集成：GitHub Actions

- Claude Code 官方就支持接 **GitHub Actions** 和 **GitLab CI/CD**。
- 几个典型用法:**PR 一来自动代码审查**（放出几个 agent 把整个代码库扫一遍,专抓逻辑错误、安全漏洞、改坏了的旧功能）、**issue 自动修复**、再用 schedule 触发器,搞**真正不靠你电脑、人走照跑**的定时自动化。
- 自带一个 **`/code-review`** 技能:派个干净的子代理,专门审你当前这次改动（diff）,审完把发现直接送回会话。

## 7.7 多会话并行 & Agent Teams：真正的"一支队伍"

**四种并行打法（看你想亲自盯多紧，挑一种）：**

1. **Git Worktrees** —— 几个 CLI 会话,各待在各自隔离的检出目录里干活,谁也不踩谁。
2. **桌面端** —— 一个面板看住多个本地会话,每个自带一份 worktree。
3. **Claude Code on the web** —— 跑在 Anthropic 云端的隔离 VM 里。
4. **Agent Teams** —— 让它们自己协调:共用一份任务清单、彼此互发消息,还有个"队长"统筹全局。

**写手 / 审查员 双会话模式（最实用的一种并行）：** 一个埋头写,另一个专门挑刺,互不护短。

| 会话 A（写手） | 会话 B（审查员） |
|---|---|
| "给 API 端点实现一个限流器" | |
| | "审查 @src/middleware/rateLimiter.ts 的限流实现。找边界情况、竞态条件、和现有中间件模式的一致性问题" |
| "这是审查意见：[B 的输出]。逐条解决" | |

为什么这招灵?**会话 B 上下文干干净净,那段代码又不是它写的,它就不会护着、不会觉得"我写的肯定没错"。**（这正是模块四里"找个第二意见"的老道理。）同理,你也能让一个会话先写测试,另一个会话写代码去把测试跑过。

**对抗性审查（无人值守跑得越久，这越要紧）：**

```text
用子代理对照 PLAN.md 审查限流器的 diff。
检查每条需求都实现了、列出的边界情况都有测试、
没有改动任务范围之外的东西。只报告缺口，不报告风格偏好。
```

> ⚠️ 注意:你让一个审查者去"挑刺",它**总归会挑出点什么来**——这是它的本能。所以一定得把话说死:**只许标那种真影响正确性、或没满足既定需求的缺口**。不然你会被它带进过度工程的坑——多余的抽象层、防御过头的代码、为根本不会发生的情况硬写一堆测试。

## 7.8 自动化的安全边界（务必读完）

无人值守 ≠ 撒手不管。安全的无人值守,是三样东西缺一不可:**先给好"尺子"（验证闭环，模块四）+ 系好"安全绳"（工具白名单 `--allowedTools`）+ 圈好"围栏"（沙箱或容器）。**

- `--dangerously-skip-permissions` 等于把所有审批一关了之、全程绿灯放行：**只建议在"断了网的容器"里用**,不然,轻则数据丢失,重则被人塞坏指令(提示词注入)牵着鼻子走。
- 那架梯子再啰嗦一遍:**交互模式跑顺 → 封装 Skill → 接 hook / 定时 → 最后才完全无人值守。** 一级都别跳。
